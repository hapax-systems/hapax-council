# Non-Anthropomorphic Segment Prep Framework Stack

**Status:** current_authority
**Checked at:** 2026-05-06T19:05:39-05:00
**Scope:** content prep, segment quality, actionability, layout responsibility, source recruitment, and prep-resumption gates.
**Authority kind:** design decision and implementation gate.

## Decision

Segment prep is governed by an operational communication stack, not by a metaphysical theory of machine personhood. Bogost and Latour remain secondary research lenses only. Tononi/IIT is removed from this work and must not appear in operational prompts, prep docs, or implementation comments unless the operator explicitly reopens that topic.

The operational foundation is a consulted standards surface, not prompt-facing doctrine. Prompt text may receive a compact role brief and required output contract, but framework names and slogans are review/development tools, not material for generated prose:

1. **Grounded contribution protocol** - a spoken contribution is only presented until source, action, layout, or audience/readback evidence makes it grounded enough for current purposes.
2. **Situated action and runtime authority** - plans, scripts, rubrics, and layout intents are resources for action, not execution commands.
3. **Source consequence** - a source must change claim shape, confidence, scope, comparison, action, or refusal. Source labels alone do not count.
4. **Argument audit** - substantive claims need claim, grounds, warrant, qualifier, and failure/rebuttal pressure.
5. **Visible or doable actionability** - what is said should map to something seen, changed, compared, ranked, polled, cited, or explicitly bounded as spoken argument.
6. **Layout responsibility** - default/static layout is not responsible success for Hapax-hosted segments. Prepared artifacts propose typed layout needs; runtime readback decides.
7. **Segment contract** - every role has production obligations, exemplars, counterexamples, quality bands, and a live-event standard. These are consulted standards, not rigid universal templates.
8. **Marked analogy discipline** - analogy is valid when the mapped relation, anchor, and limit are explicit.
9. **Doc-rot control** - existing docs are evidence until checked, not authority by age or polish.
10. **Control-loop admissibility** - systems/control language is valid only when the loop boundary is explicit. Prepared artifacts may emit `feedforward_plan` loop cards; runtime components may claim `closed_loop` only when sensors, actuators, timing, fallback, and readback exist. Canonical map: `docs/superpowers/specs/2026-05-06-systems-control-segment-prep-runtime-map.md`.

## Why This Stack

The chosen stack follows the design commitments better than a Bogost/Latour master frame because it maps directly to the failures we have to prevent:

- Generic scripts fail because they lack premise, source consequence, and role-specific production obligations.
- Good-sounding but inert scripts fail because spoken claims have no visible or doable counterpart.
- Default layout fails because it is a static crutch in a context where Hapax is responsible for content and presentation quality.
- Anthropomorphic prep fails because it borrows human-host personality, feeling, empathy, concern, belief, and taste language instead of describing actual system operations.
- Citation bulk fails because it can decorate a segment without changing its claims.
- Rigid rubric worship fails because professional standards are resources for judgment, not an expert-system replacement for judgment.

## Framework Status

Primary operational frames:

- Grounded contribution protocol: Clark/Brennan and Traum are the research lineage for presentation, evidence, repair, and acceptance.
- Situated action/EMCA: Suchman and conversation analysis are the lineage for plans as resources and runtime circumstances as authority.
- Source consequence/provenance: W3C PROV, epistemic vigilance, and editorial standards are the lineage for traceable evidence and trust pressure.
- Argument audit: Toulmin is a compact claim/grounds/warrant/failure check, not a visible template every segment must recite.
- Segment contract: broadcast show prep, role-specific craft, professional vision, boundary objects, examples, and counterexamples provide the working standards.
- Actionability/layout observability: HCI affordance/feedback and observability/control-loop practice provide the visible/doable/readback standard.

Secondary only:

- Bogost/Alien Phenomenology: useful for anti-anthropocentric vocabulary and artifact-making discipline. It does not authorize claims about Hapax inner life, alien consciousness, withdrawal, or object metaphysics.
- Latour/ANT: useful for mediation and black-box audits. It does not own truth, agency, grounding, source adequacy, or runtime authority.
- Distributed cognition/CSE: valid research lineage for distributed work. Prompt-facing language must translate it as multi-aperture coordination, distributed work surface, or readback-mediated coordination. Do not describe Hapax as a distributed mind.

Rejected:

- Tononi/IIT and consciousness theories.
- CASA, ELIZA, companion-agent, empathy, attachment, or relational-agent framings as design goals.
- Jungian persona/archetype and human-host personality protocols.
- Unbounded OOO/alien metaphysics.
- ANT as a master frame.
- Predictive-processing/FEP/enactivist master vocabularies for prep.
- Pure control/observability as the whole framework.
- Any framework that makes Hapax sound like it thinks, feels, perceives, wants, hopes, fears, deserves empathy, or has human concerns.

## Prompt-Facing Language

Use:

- "the segment states"
- "the prep artifact proposes"
- "the runtime readback accepts, repairs, or withholds success"
- "source pressure changes the claim"
- "the role contract requires"
- "the system has a selection prior"
- "the beat maps to a visible/doable effect"
- "by analogy, X relation maps to Y function; the limit is Z"

Avoid in active prep prompts:

- "Hapax feels", "Hapax wants", "Hapax believes", "Hapax thinks", "Hapax cares"
- "your feelings", "your show", "your opinions", "your affect", "your personality"
- "operator-flavoured", "operator-coloured"
- "Hapax's voice" as a personality surface
- "distributed cognition" as a prompt-facing self-description
- "alien mind" or any inner-life metaphor

## Segment Prep Contract

Every responsible live segment must leave these reviewable surfaces as explicit artifact fields. Deterministic derivations are replay/freshness checks only; they do not substitute for the prep artifact accepting responsibility for the claim/source/action contract.

- `rundown_card`: topic, role, premise, planned duration, beat list, source basis, expected action surfaces, fallback posture.
- `role_template_ref`: the consulted role obligations and examples.
- `claim_map`: claim, grounds, warrant, qualifier, rebuttal/failure pressure.
- `source_packet_refs`: concrete sources or explicit source-gap decision.
- `actionability_map`: spoken trigger, object, operation, expected effect, feedback/readback.
- `layout_need_map`: typed needs only, no layout command.
- `readback_obligations`: what runtime evidence would count.
- `loop_cards`: admissibility cards declaring whether control language is a feedforward prior, supervisory gate, closed loop, or analogy only.
- `segment_live_event_report`: whether speech, source object, visible/doable transformation, audience job, temporal coupling, and payoff form an actual livestream bit.
- `repair_decision`: rewrite, narrow, recruit, demote to spoken argument, switch role, or quarantine.
- `segment_loadability_report`: why this artifact is loadable or why it is quarantined.

This contract must strengthen judgment without turning into an expert rule system. A compliant but generic artifact still fails. A strong artifact with one missing source/action/readback obligation should be narrowed, repaired, or quarantined rather than polished past the gap. Validator rewrite phases are forbidden at release review: validation can reject, narrow, quarantine, or request another prep iteration, but it cannot silently become an authoring pass.

## Marked Analogy Contract

Analogy is allowed and useful for communicating across unlike entities. It must be marked as a relation transfer, not an identity claim:

```json
{
  "analogy_term": "trial",
  "source_domain": "legal procedure",
  "target_function": "claim under evidence burden with rebuttal pressure",
  "required_anchor": "declared source ref, validator, or runtime readback",
  "forbidden_extension": "judge persona, fairness feeling, moral intuition"
}
```

Allowed form: "By analogy, this beat works like a trial: the packet supplies grounds, the validator supplies rebuttal pressure, and runtime readback decides what is visible."

Forbidden form: "Hapax judges fairly because it has no bias."

## Doc-Rot Policy

Authority order:

1. Current code and passing tests.
2. Machine-readable schemas/config.
3. Current authority docs with checked dates.
4. Revalidated historical docs.
5. Historical docs without revalidation.
6. Chat context: never authority until written into repo docs.

Each durable doc that affects prep should be marked as one of:

- `current_authority`
- `candidate_reference`
- `historical_context`
- `superseded`
- `stale_risk`
- `stale_do_not_use`

Operational claims expire fastest: model names, source counts, prompt behavior, schema fields, timers, service names, branch status, context provider behavior, and current external facts must be checked against code, tests, service state, or fresh source receipts.

## Prep-Resumption Gate

Prep may resume only when all are true:

1. This framework decision exists in repo docs and is referenced by active prep docs or tests.
2. Tononi/IIT is absent from operational prompts.
3. Bogost/Latour are secondary only.
4. Prompt-facing language does not anthropomorphize Hapax.
5. Active prep prompts preserve source consequence, role contract, actionability, and proposal-only layout.
6. Default/static layout cannot certify responsible live segment success.
7. Command-R residency remains fail-closed with no Qwen, no unload/swap, and no fallback route.
8. Eligible candidates are not runtime pool content until a separate `selected-release-manifest.json` is written from candidate-set review.
9. One canary segment must pass automated review plus team critique before continued pool release.
10. Existing docs that may mislead future sessions are marked current, historical, superseded, or stale-risk.
11. Selected-release feedback must publish selected artifacts and review precedent only after the selected loader gate accepts them; future prep may consult the digest as prior-only evidence, not runtime authority. Publication failure is a diagnostic on future-prep memory, not a reason to revoke a valid selected manifest.
12. Candidate-set review must rank all eligible artifacts before release selection. The top-N release window requires excellence receipts; below-cutoff review gaps are diagnostic.
