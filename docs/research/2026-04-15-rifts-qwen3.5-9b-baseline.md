### 2.3 RIFTS accuracy scores — harness limitation disclosure

## 4. Divergence analysis — where benchmark ≠ live behavior

**Template thesis:** A good benchmark predicts live behavior. A poor benchmark measures something systematically different from production. This section enumerates the divergence vectors hapax can test.

- **Gap:** **[FILL: if RIFTS is systematically shorter/longer than live, call it out]**

### 4.2 System prompt divergence

- **RIFTS harness sends no system prompt.** The chat completion is a single user-role turn with just the RIFTS `instruction` field.
- **Live voice loop attaches a ~2000-token persona system prompt** (`agents/hapax_daimonion/persona.system_prompt`).
- **Implication:** the RIFTS baseline measures Qwen3.5-9B's raw grounding behavior, not Qwen3.5-9B + hapax-persona. The persona system prompt may bias responses toward more/fewer clarifying questions in a way RIFTS cannot detect.
- **Compounding concern:** the production `max_seq_len: 4096` + ~2000-token persona leaves only ~2000 tokens for operator turn + response. Any live turn exceeding that budget gets truncated. See queue #229 closure inflection (2026-04-15T20:11:30Z) + proposed #231 for the raise-to-16K+ recommendation.

### 4.3 Multi-turn divergence

- **RIFTS is single-turn.** Each prompt is an isolated chat completion.
- **Live voice loop is multi-turn.** Grounding acts happen over N turns; RIFTS cannot measure the "operator clarifies, model proceeds on second turn" path.
- **Implication:** RIFTS may underestimate hapax's effective grounding because it doesn't give the model the conversational context that unlocks clarification-then-answer sequences.

### 4.4 Temperature divergence

- **RIFTS harness: temperature=0.0** (deterministic, for reproducibility)
- **Live voice loop: per-route temperatures** (~0.7 for conversational, 0.1 for coding/reasoning)
- **Implication:** live responses are more stochastic; RIFTS baseline is a lower bound on response-variance.

### 4.5 Thinking-mode divergence

- **RIFTS harness: thinking mode disabled** (v2 §2.1 E1: local-fast has thinking off in production, and the harness mirrors production)
- **Paper's frontier baselines presumably included thinking mode** (the paper is published before Qwen3.5 thinking was widely toggled off in production deployments)
- **Implication:** hapax's baseline is NOT directly comparable to the paper's 23.23% frontier number on a thinking-mode-apples-to-thinking-mode-apples basis. The comparison is "hapax production config" vs "paper frontier config". Document this explicitly; do not treat the 23.23% number as a same-substrate comparison target.
