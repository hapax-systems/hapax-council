### Part 2 — ARCHITECTURE

Three asks, one principle: **everything below is connect-the-executor work on already-built, principled machinery — not new contracts, not dB tables, not dispatchers.** Each item is classified buildable-now / gated / unknown.

#### (a) Ideal music levels + ducking — the principled mechanism (no dB rule-table)

The design is already no-presets-compliant (LUFS-derived constants + RMS envelope sidechain + deepest-duck arbitration + ramps). **The executor is just pointed at a deleted node.** Three layers:

1. **Restore a live gain surface in the music→tap path (buildable-now).** The mk5 topology has no controllable node between `hapax-music-loudnorm-playback` and `hapax-livestream-tap`. Either (a) re-insert one controllable-gain filter-chain node there, or (b) have the daemon modulate `hapax-music-loudnorm` node volume directly via `wpctl`/`pw-cli`. **Option (b) is correct** — fewer moving parts, matches the mk5 single-gain reality, and `audio-topology-reference.md:92,117` *already documents this as the behavior* (make code match doc). Re-target `MUSIC_DUCK_NODE` accordingly. Also re-target the daemon's operator-VAD input off the dead L-12 AUX4 to the live operator mic node (the bilateral fix), or it stays half-inert.

2. **Key it on a real `tts_active` signal (gated on item-b below).** The duck must read `tts_active` from `voice-state.json`. That key is absent live because `TtsStatePublisher` only fires on the pipecat path, which the live daimonion doesn't run. Publish `tts_active` from the *hosting* path (CPAL/director broadcast speech), then bind either the existing daemon or instantiate `TtsDuckController` in the compositor boot path against the live music gain. Depths stay sourced from `audio_loudness.py` — **−8 dB TTS, −12 dB operator, deepest-duck wins, 10 ms attack / 400 ms release.**

3. **Close the LUFS loop for sustained hosting (buildable-now).** Replace the open-loop `+16 dB` makeup with a slow ebur128-integrated controller on `hapax-broadcast-normalized` that nudges master makeup toward `EGRESS_TARGET_LUFS_I=−14` (long time-constant, bounded ±dB). This is what makes multi-minute segments land on target regardless of source mastering — the *sustainable* mechanism vs the current −8 to −10 dB shortfall.

4. **Hosting-mode hold (buildable-now).** For a paced spoken span the 200 ms VAD hold-open un-ducks between sentences. Drive the duck off a **segment-active** signal the CNS already owns (the segment beat state), holding the bed at −8 dB for the whole span and releasing only at segment end — not a per-utterance timer. This is a subscription, not a rule.

This is principled because depth is a LUFS-relative offset + measured speech-over-music delta, and absolute loudness is carried by pre-norm + the closed master loop — never a per-segment dB schedule.

#### (b) TTS ready for sustained hosting

1. **Wire the host onto the existing broadcast path (buildable-now, highest leverage).** Change `director_loop._play_audio` to request `VoiceOutputDestination.PUBLIC_BROADCAST` **through `resolve_playback_decision`'s fail-closed gates** (intent + ≤120 s programme auth + audio_safe + tts_permission), instead of hard-coded `private_monitor`. Keep `private_monitor` as an additive fallback (never-remove). A naive sink swap that bypasses the classifier would risk leaking private narration — the gate is the invariant.

2. **Gapless delivery (buildable-now).** Convert hosting from per-clause spawn + `time.sleep` to a **single persistent pw-cat stream fed a sentence queue with look-ahead synthesis** (synthesize sentence N+1 while N plays). Keep clause chunking as the *prosody* unit, not the *stream* unit. This eliminates inter-clause seams and absorbs Chatterbox GPU / Kokoro CPU jitter — the structural fix for multi-minute arcs.

3. **Arc-level prosody (buildable-now).** Pass segment role + position-in-arc into the synth request so loudness/rate/emphasis are planned across the span, not clause-local.

4. **Reactivity is already correct (wired-live, preserve).** The `tts-envelope.f32` + `speech-wave.bin` rings tap PCM via the write-wrapper and feed the Sierpinski-centre oscilloscope; the director path already routes through `_play_pcm_with_envelope`. No change — just don't break it when the stream becomes persistent.

**Unknown:** whether Chatterbox is actually GPU-resident in the running daimonion or fell back to Kokoro CPU (changes real synth latency / look-ahead depth needed). Verify GPU residency before sizing the look-ahead buffer.

#### (c) In-CNS director-move orchestration — one model

The correct substrate already exists and is partly live; the spec defers "livestream segments" to §10 LATER, but **segment hosting is the operator's NOW**, so build the segment instance next, reusing the live put-side rather than waiting on drift/compositing PRs. **Generalize the live layout-responsibility loop from "which posture" to "which surface action" — do not add a parallel path.**

Authored directorial requirements → CNS-orchestrated surface actions, one flow:

> A segment beat's already-authored `layout_need_map` / `loop_cards` / `beat_action_intents` / `segment_assets` (proposal-only, `declares_layout_needs_only`, forbidden from naming concrete commands/coords) become **typed `SurfaceActionNeed`s** → the **same pure controller** decides which bounded surface action satisfies each need → dispatched through the **one live Via** (`AffordancePipeline.select()` → `compositional_consumer.dispatch_*`) → applied → **counted as success only when a witnessed readback proves the visible effect.**

Concrete closures, each reusing existing machinery:

- **The missing materializer (buildable-now, single highest-leverage move).** Build the executor that reads `active-segment.json`'s declared needs and emits `current_beat_action_intents` / `BeatLayoutIntent`s so the runner stops refusing with `no_layout_needs`. **It must emit typed needs that `select()` recruits against parametrically — NOT `if need==worked_example: show_card`.** That `if/elif` would re-introduce the exact brittle dispatcher the operator is killing. Everything downstream is already wired and waiting on this one hop.
- **play-YT-on-OARB (buildable-now + gated):** add a `media.cue_to_surface(media_ref → OARB slot)` action to `dispatch_youtube_direction`'s vocabulary (it currently can't carry a target) and either teach `_honor_youtube_direction` to load the specific ref or route to the live `screwm-oarb-playlist-rotator` `/play` POST. Audio **must** inherit the existing slot audio-control (YT channel + cut-away muting) — never open a new playback path (L-12/topology invariant). **Gated:** arbitrary URLs stay behind fortress/consent; within the operator-curated set, no new approval.
- **show-image (buildable-now):** give `segment-playback.json` a real renderer — decode + blit the image via the existing `asset_front` posture / pip-draw path; route youtube assets to `media.cue_to_surface`. Replacing the `[IMAGE] caption` text label with the actual shown media *fixes the gap and better honors show-don't-tell* — do not prettify the label.
- **show-X / highlight-Y (buildable-now):** route authored `beat_action_intents` (show_evidence/compare/cite) through the controller into the **existing** `overlay.emphasis` / `ward.highlight` / `gem.*` dispatchers, so prepared segments deterministically get their highlights while the narrative director still adds spontaneous ones (additive, never-remove).
- **Unify, don't duplicate (buildable-now):** map `segment_layout_contract`'s `RuntimeReadbackRef` + `LayoutReceiptStatus` (matched/mismatched/fallback) onto the Chiasm **get** — readback *is* prediction-error ("did the surface realize what was expressed?"). `may_command_layout=Literal[False]` already *is* "get is a sensor, never a kill-switch." Extend the readback to assert media `object_ref`s rendered (image blitted, OARB slot playing the intended ref) — so a media move can't fake-succeed. Resolve both SHM buses (`active-segment.json` layout + `segment-playback.json` assets) by one controller keyed on `programme_id + beat_index`; keep the two writers, unify only the resolver.
- **Consent thread (gated, must activate before media re-broadcast):** activate `agentgov` `Labeled[A]` on the segment value channel before any recruited YT/image reaches broadcast — it is the principled (not expert-rule) carrier of protected/PRIVATE provenance through re-perception. Wrap media moves in fortress/working-mode egress gating.
- **Loop-gain guard (gated):** hold the spec's PR-E feedback-gain measurement before closing get→select() write-back, so a re-perceived "show image" can't re-recruit another.

**Unknowns:** (1) whether the live recruitment families are driven by segments today or only the ambient DMN/imagination loop; (2) whether the materializer code is absent, disabled, or failing silently (needs a read of `programme_loop.py`'s segment-emit path); (3) whether OARB authored-ref playback should preempt the curated rotation or target a separate mount; (4) whether the operator wants arbitrary YT URLs at all vs strictly curated-set references.

**Invariant compliance.** L-12 invariant is *stale* — live hardware is MOTU mk5 + Torso S-4, no Zoom/MPC card present; honor the spirit (operator-owned hardware, all fixes software-side) and do not propose reviving L-12/MPC routing. Protected-audio/private: every host-voice and media move goes through `resolve_playback_decision` + agentgov + fortress gating; the duck targets content sinks only, never the operator path. Show-don't-tell: moves act on surfaces, never narrate; replacing the asset text-label with real media advances this. No-presets: levels = LUFS targets + closed loop + envelope sidechain; moves = typed needs recruited parametrically. Never-remove: ducker daemon, duck constants, role-duck, master limiter, `private_monitor`, DURF text fallback, and `segment_layout_contract` are all **extended, not deleted**. Fortress: broadcast voice + ducking + media egress remain subordinate to stream-mode gating (`get_stream_mode_or_off` defaults OFF).

**The through-line:** none of this is new infrastructure. The egress chain runs; the loudness SSOT is principled; the recruitment Via runs; the readback discipline is the right model; the segment contract already authors the directorial requirements. The system has repeatedly shipped the contract and skipped the executor — so the entire solution is **connect the dormant executors to the live substrate**: re-target the ducker to a real node + close the LUFS loop; route the host onto the broadcast path with a persistent stream; build the one missing needs→action-intents materializer that feeds the existing controller + recruitment + readback. That is what makes it RUN rather than become another dormant formalism.

## Critic — bandaid / expert-rule / invariant / spec-not-executor / unverified / missing

### Bandaids
- Forward-direction (a)(1) option (b) 'modulate hapax-music-loudnorm node volume directly' co-opts the per-source true-peak LIMITER node (Limit=-18 dB, fast_lookahead_limiter_1913) as the duck gain surface. Riding a limiter's output volume conflates two concerns (loudness normalization vs sidechain ducking) on one node; it is the path of least resistance, not the clean fix. It only looks principled because audio-topology-reference.md happens to document it ('make code match doc') -- but the doc itself is drift, not a design decision. A dedicated controllable-gain node in the music->tap path (option a) is the non-bandaid choice; the synthesis leans (b) for fewer-moving-parts, which is bandaid reasoning.
- Re-targeting the audio_ducker is presented as 'the smallest fix, restores ducking' -- but the verified corpus shows the breakage is BILATERAL: the operator-VAD INPUT it reads is the dead L-12 AUX4 node (rode.rms=-119 dBFS, digital floor). The synthesis acknowledges this in passing ('or it stays half-inert') but the headline 'connect-the-executor' framing understates that BOTH the input tap and the output target are dead. Fixing only the output node is a bandaid that leaves operator-voice ducking inert; the full fix must re-source the VAD input too, and that is not a one-line re-target.
- Routing the new media.cue_to_surface to the live screwm-oarb-playlist-rotator's /play POST (an alternative offered in (c) play-YT-on-OARB) bolts the director onto a script that was explicitly built to REPLACE the retired director's rotation. Reusing it as a director actuation channel re-couples the two and risks the rotator and the director fighting over the same OARB slot -- a bandaid that papers over the retirement rather than designing a single owner of the OARB media slot.

### Expert-rule / presets risk
- The segment action-intent materializer (the single highest-leverage build) is the prime preset/expert-rule risk and the synthesis itself flags it: a naive 'if need==worked_example: show_card' / 'if need==play_yt: load_video()' need->move lookup table would re-introduce exactly the brittle dispatcher the operator wants killed. The flag is correctly raised, but the synthesis does NOT specify the parametric recruitment mechanism concretely enough to guarantee the implementer avoids the table -- 'emit typed needs that select() recruits against parametrically' is asserted, not designed. Without a worked example of how a 'worked_example' need maps to a recruited capability via ModTargets/affordance scoring rather than a switch, the materializer will almost certainly land as an expert-rule table.
- Adding a media.cue_to_surface ACTION to dispatch_youtube_direction's vocabulary risks growing the typed-dispatcher set into a per-move command catalogue (the ~33-family / ~35-dispatcher sprawl the CNS is supposed to collapse). Each new bespoke action verb (cue_to_surface, asset_show, ...) added to compositional_consumer is a step toward the brittle command dispatcher unless it is genuinely recruited parametrically rather than branched on by family name.
- The proposed 'hosting-mode hold' that keeps the bed ducked for the whole spoken span risks becoming a fixed-duration / fixed-state cadence rule. The synthesis says to drive it off a CNS-owned 'segment-active' signal (good), but if that signal is unavailable (see unknowns) the fallback temptation is a hardcoded hold timer -- a cadence preset that violates no-expert-rules.

### Invariant violations (incl. LIVE)
- Show-don't-tell, current LIVE violation (not just a proposed risk): coding_activity_reveal.py:778 renders authored assets as the literal on-screen text '[IMAGE] caption' / '[YOUTUBE] url'. This is a label that textually narrates the director's intended action -- already a show-don't-tell breach in the running renderer. The synthesis correctly notes replacing it with real media fixes this, but it should be flagged as a CURRENT invariant violation, not merely a dormant gap.
- Protected/PRIVATE -> broadcast: re-routing director_loop._play_audio from private_monitor to public_broadcast is the highest-risk change in the whole plan. The synthesis correctly requires going through resolve_playback_decision's fail-closed gates -- but a naive sink swap (the tempting minimal diff) bypasses the classifier (intent + <=120s programme auth + audio_safe + tts_permission) and leaks operator-private narration to the livestream. This must be enforced, not trusted; the gate is the invariant boundary.
- Protected media egress: activating media.cue_to_surface for arbitrary YT URLs opens an un-consented egress channel. agentgov Labeled[A] (the principled provenance carrier) is built but imported NOWHERE in the binding path -- so today there is no live consent thread guarding recruited media. Shipping media moves before activating that gate, or relying solely on the operator-curated set without the agentgov thread, risks protected-content leakage and fortress-bypass.
- L-12 invariant is STALE and must NOT be honored literally: live hardware is MOTU UltraLite mk5 + Torso S-4, no Zoom L-12 / Akai MPC card present (pactl). The risk is a fix that 'restores' ducking by reviving the L-12-era hapax-music-duck node or L-12/MPC routing -- that would chase decommissioned hardware. Honor the spirit (software-side fixes only); reject any proposal that re-introduces L-12/MPC nodes.
